## 一、專案概述

- **名稱**：智慧驗證 tester (Intelligent Validation Tester)
- **用途**：O-RAN 架構下的資料與 AI 效能驗證儀表板
- **目標**：驗證受測平台 (DUT) 的資料準確性與 AI 應用的智慧程度，支援 benchmark 比較
- **主要使用者**：平台廠商、應用廠商、驗證管理員、檢視者
- **對外公開**：是（首頁/產品頁採 SSR，儀表板需登入）
- **響應式**：全頁支援（手機 / 平板 / 桌機）

## 四、架構哲學：三層分離

前後端共用的架構哲學：

```
前端                           後端
─────────────                 ─────────────
Component (UI)                View (DRF ViewSet)
     ↓                             ↓
Hook (Logic)                  Service Layer
     ↓                             ↓
Service (API Client)          Model (ORM) + Connector
     ↓                             ↓
──── HTTP/WebSocket 契約 ────────────
```

**核心規則**：
- 每層**單一職責**，嚴禁跨層混合
- 所有外部系統（O-RAN、攝影機）走**可插拔 Connector**（Mock / 實機切換）
- 前端 Component / 後端 View 都是**薄控制器**，業務邏輯集中在 Hook/Service

詳細規則與範例請參考 02、03 文件。

### 資料流示例:啟動一次驗證

**目前實作(同步 mock)**

```
① 使用者點「執行驗證」 → RunValidationDialog 開啟
② 從 baseline.test_scenarios_detail 中選一份情境(資料集)
③ 前端 Component
   └─► Hook (useDataValidationRun)
       └─► Service (axios POST /api/dut/{id}/run-data-validation/, body = {scenario_id})

④ Django 接收請求
   └─► View (DutViewSet.run_data_validation action)
       ├─► 驗證 scenario_id 在 baseline.test_scenarios M2M 中
       ├─► 隨機產生 mock metrics(completeness/accuracy/timeliness)
       └─► 同步回傳 {scenario, passed, score, metrics, thresholds}

⑤ 前端拿到結果
   └─► ValidationResultCard 顯示情境名稱 + 分數 + 三個指標(每個對照門檻判 pass/fail)
```

**未來真實驗證引擎(Celery + WS,尚未實作)**

```
① 使用者點「開始驗證」
   └─► Service (axios POST /api/validation-runs/)

② Django 接收請求
   └─► ValidationService.start_run
       ├─► 寫入 ValidationRun(pending)
       └─► 派遣 Celery Task

③ Celery Worker 執行
   ├─► ValidationStrategy.run()
   │    └─► Connector (O1/A1 Mock 或實機)
   ├─► 每步 publish 到 Channel Layer (Redis)
   └─► Channel Layer → Django Channels Consumer
        └─► WebSocket → 前端 useRunStream Hook
             └─► Component 即時更新進度

④ 完成後
   ├─► 寫入 Metric(Timescale)
   ├─► 產生報告 PDF(WeasyPrint → MinIO)
   └─► WS 推送 'completed' 訊息
```

> 兩條流程**並行存在**:`run-data-validation` 是目前演示用的同步 mock;真正接 connector 後改用 ValidationRun 流程。當前 ValidationRun model 與 channels consumer 已就緒,strategies/ 目錄留 stub 等實機介接。

## 十一、部署與維運

### 11.1 docker-compose 服務清單

本專案以**單一 `docker-compose.yml`** 一鍵啟動所有服務：

| 服務 | 容器名 | 端口 | 說明 |
|------|-------|------|------|
| `frontend` | ivt-frontend | 3000 | Next.js (`next start`) |
| `backend-web` | ivt-backend-web | 8000 | Django + DRF（Gunicorn WSGI）|
| `backend-ws` | ivt-backend-ws | 8001 | Django Channels（Daphne ASGI）|
| `celery-worker` | ivt-celery | — | Celery Worker（共享後端映像）|
| `celery-beat` | ivt-celery-beat | — | Celery Beat（定時任務）|
| `postgres` | ivt-postgres | 5432 | PostgreSQL 16 + TimescaleDB |
| `redis` | ivt-redis | 6379 | Celery broker + Channel Layer |
| `minio` | ivt-minio | 9000, 9001 | 物件儲存（API + Console）|
| `mediamtx` | ivt-mediamtx | 8554, 8888 | RTSP 接收 + HLS/WebRTC 輸出 |
| `nginx` | ivt-nginx | 80, 443 | 反向代理 |

### 11.3 Nginx 反向代理

```nginx
# deploy/nginx.conf
upstream frontend { server frontend:3000; }
upstream backend_web { server backend-web:8000; }
upstream backend_ws { server backend-ws:8001; }
upstream mediamtx { server mediamtx:8888; }

server {
    listen 80;
    server_name localhost;

    # Next.js（含 SSR）
    location / {
        proxy_pass http://frontend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # Django REST API
    location /api/ {
        proxy_pass http://backend_web;
        proxy_set_header Host $host;
    }

    # Django Admin
    location /admin/ {
        proxy_pass http://backend_web;
    }

    # Django Channels (WebSocket)
    location /ws/ {
        proxy_pass http://backend_ws;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }

    # MediaMTX HLS
    location /hls/ {
        proxy_pass http://mediamtx/;
    }

    # MinIO（報告下載）
    location /minio/ {
        proxy_pass http://minio:9000/;
    }
}
```

### 11.5 啟動順序

### 11.6 日常維運指令

| 操作 | 指令 |
|------|------|
| 啟動全部 | `docker compose up -d` |
| 停止全部 | `docker compose down` |
| 重建後端 | `docker compose up -d --build backend-web backend-ws celery-worker` |
| 進入 Django shell | `docker compose exec backend-web python manage.py shell` |
| 執行 migration | `docker compose run --rm backend-web python manage.py migrate` |
| 建立 superuser | `docker compose run --rm backend-web python manage.py createsuperuser` |
| 收集靜態資源 | `docker compose run --rm backend-web python manage.py collectstatic` |
| 備份 DB | `docker compose exec postgres pg_dump -U ivt ivt > backup.sql` |
| 還原 DB | `cat backup.sql \| docker compose exec -T postgres psql -U ivt ivt` |
| 清空全部重來 | `docker compose down -v`（⚠️ 會刪 volume）|

### 11.8 監控與日誌（初期）

- **日誌**：`docker compose logs -f <service>`（stdout/stderr 由 Docker 蒐集）

### 11.9 安全加固（上線前）

| 項目 | 做法 |
|------|------|
| HTTPS | Nginx 加 Let's Encrypt 或自簽憑證 |
| SECRET_KEY | 改為強隨機字串（不要用 `change-me`）|
| DB 密碼 | 改為強密碼，`.env` 不進版控 |
| MinIO | 改掉 `minioadmin` 預設帳密 |
| Django DEBUG | `DEBUG=False`，設定 `ALLOWED_HOSTS` |
| Rate Limiting | Nginx limit_req + django-ratelimit |
| 防火牆 | 只開 80/443，其他端口對外關閉 |
| 初始管理員 | 首次登入強制改密碼 |
| CORS | 白名單正式網域 |

### 11.10 擴充路徑

當流量增大時：

| 瓶頸 | 解法 |
|------|------|
| Django 處理不完 | 增加 `backend-web` 副本 + HAProxy |
| WS 連線過多 | 增加 `backend-ws` 副本（Channels 自動透過 Redis 協調）|
| 驗證任務多 | 增加 `celery-worker` 副本 + 分 queue |
| DB 讀多寫少 | PostgreSQL Read Replica |
| 報告檔量大 | MinIO 換雲端 S3 |
| 攝影機多 | MediaMTX 水平擴展 |
| 整體升級 K8s | docker-compose.yml 可轉換（kompose 工具）|
