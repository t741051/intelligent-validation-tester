> 適用範圍:所有要在同一座電視牆上展示的前端專案。本文件由 IVT(智慧驗證 tester)專案率先實作整理而成。

## 一、展示環境硬體規格

| 項目 | 規格 |
|------|------|
| 螢幕拼接 | **6 (橫) × 3 (縱) = 18 顆** |
| 單顆螢幕 | 1920 × 1080 (FHD) |
| 拼接後總解析度 | **11520 × 3240** |
| 縱橫比 | 32 : 9 |
| 訊號源 | 單條 HDMI(主機輸出 11520×3240) |
| 邊框分隔位置 | 直線 x = 1920, 3840, 5760, 7680, 9600(共 5 條)<br>橫線 y = 1080, 2160(共 2 條) |
| 邊框寬度 | 各家面板不同,設計時保留 ±50px 安全區即可 |

### 1.1 Bezel 在版面上的實際物理意義(設計師必讀)

「Bezel」= 兩台電視之間的**實體黑色邊框**,**不是 padding、不是 margin、不是 gap**。它是真實存在、看不到任何內容的物理間隙。對畫面而言:

- ❌ **不是縫隙裡有暗色區塊** — 它就是「沒有畫面」,黑得徹底
- ❌ **不會自動把內容讓開** — 瀏覽器不知道有邊框,你不主動避開,內容就會被吃掉
- ❌ **不能假設邊框很窄可以忽略** — 一行 24px 高的字若橫跨垂直 bezel,中間 1-2 個字直接消失,不是模糊、不是被切一半,**是看不到**

**設計約束(必須遵守):**

1. **任何「需要被讀」的內容遠離 bezel ±50px 安全區**
   - 不要讓單一字元、icon、按鈕剛好踩在 bezel 上
   - 表格儲存格邊界、卡片邊緣應該對齊或避開 bezel,不要任意跨越

2. **跨多顆螢幕的圖片/圖表 — 視覺上要「假裝」中間有縫**
   - 例如一張寬地圖橫跨 3 顆 TV(總寬 5760px),設計者要**預期** x=1920 跟 x=3840 那兩處會有黑線把畫面切成 3 塊
   - 不是壞事 — 觀眾離 3-5 公尺看,大腦自動「補」回去,類似看雙螢幕辦公差不多
   - 但**不能把字、按鈕、圖示放在那兩處**,會被吃掉

3. **小元件(KPI 數字、按鈕、Badge)— 完全在單顆 cell 內**
   - 1 顆 cell = 1920×1080,寬鬆夠用
   - 跨 cell 邊界的元件,要嘛直接占滿 2 顆(中間黑縫接受),要嘛縮在 1 顆內

4. **內容區塊建議用 1 顆 cell 為單位設計**
   - 像砌磚:整顆、半顆、整列、整欄都行,但**不要剛好跨 0.7 顆**
   - 卡片網格 grid-gap 至少 64px,讓卡片邊緣有機會自然停在 bezel 兩側

**簡單口訣:**

> 重要內容 → 不跨縫 / 跨大幅內容(地圖/圖表)→ 接受縫 / 邊界元件 → 對齊 cell

## 三、電視牆模式 (Wall Mode) 實作

核心思路:**邏輯畫布固定 11520×3240,JS 算 scale 把整面 body 縮到當下 viewport**。在筆電上預覽 = 自動縮成等比例小圖;真牆上 scale=1.0 自動恢復原生大小。同一份程式碼跨環境通用。

### 3.1 開關設計

- **手動 toggle**,不要用 viewport 寬度自動偵測(否則筆電上沒辦法測)
- 狀態存 `localStorage`(Zustand persist),跨頁面 / 重新整理保持
- Toggle 放 Header 右側,使用 `Tv` / `Tv2` icon

### 3.2 必要組成

| 檔案 | 作用 |
|------|------|
| `stores/wallModeStore.ts` | Zustand store,`isWall` boolean + persist |
| `components/layout/WallModeApplier.tsx` | 監聽 store,在 `<html>` 加/移 `wall-mode` class,並用 JS 計算 scale 套到 body 的 inline transform |
| `styles/globals.css` 的 `html.wall-mode {...}` 區塊 | 所有 wall-mode 才生效的樣式覆寫 |
| Header 上的 toggle button | 切換用 |

### 3.3 Zustand store 範例

```ts
// stores/wallModeStore.ts
import { create } from "zustand";
import { persist } from "zustand/middleware";

type WallModeState = {
  isWall: boolean;
  toggle: () => void;
  setWall: (v: boolean) => void;
};

export const useWallModeStore = create<WallModeState>()(
  persist(
    (set) => ({
      isWall: false,
      toggle: () => set((s) => ({ isWall: !s.isWall })),
      setWall: (v) => set({ isWall: v }),
    }),
    { name: "your-app-wall-mode" },
  ),
);
```

### 3.4 Applier(套 class + JS scale)

```tsx
// components/layout/WallModeApplier.tsx
"use client";
import { useEffect } from "react";
import { useWallModeStore } from "@/stores/wallModeStore";

const WALL_W = 11520;
const WALL_H = 3240;

export function WallModeApplier() {
  const isWall = useWallModeStore((s) => s.isWall);
  useEffect(() => {
    const root = document.documentElement;
    const body = document.body;
    if (!isWall) {
      root.classList.remove("wall-mode");
      body.style.transform = "";
      return;
    }
    root.classList.add("wall-mode");
    const apply = () => {
      const scale = Math.min(window.innerWidth / WALL_W, window.innerHeight / WALL_H);
      body.style.transform = `scale(${scale})`;
    };
    apply();
    window.addEventListener("resize", apply);
    return () => window.removeEventListener("resize", apply);
  }, [isWall]);
  return null;
}
```

> **為何不用 CSS `transform: scale(min(calc(100vw/11520), ...))`** — `100vw/11520`(length÷number)結果還是 length,不是 `scale()` 需要的 unitless。CSS `min(length, length)/length` 在多數情境工作,但 JS 計算最穩、也最容易擴展。

### 3.5 全域 CSS 骨架

```css
/* 1. 畫布尺寸 */
html.wall-mode {
  font-size: 48px;        /* 3× 一般模式,所有 rem 自動放大 */
  overflow: hidden;       /* 阻止 oversized body 產生 scrollbar */
}

html.wall-mode body {
  width: 11520px;
  height: 3240px;
  min-width: 0;
  min-height: 0;
  transform-origin: top left;
  /* transform: scale(...) 由 WallModeApplier 動態設定 */
}

/* 2. 全屏 chrome:覆寫 h-screen 讓 AppShell 吃滿整個邏輯畫布 */
html.wall-mode .h-screen {
  height: 100% !important;
}

/* 3. 文字最低尺寸:小字 (text-xs/sm/base) 在 wall mode 都至少 2rem (96px),
   line-height 也要一起拉,否則上下行會重疊 */
html.wall-mode .text-xs,
html.wall-mode .text-sm,
html.wall-mode .text-base {
  font-size: 2rem !important;
  line-height: 1.4 !important;
}
html.wall-mode .text-lg {
  font-size: 2.25rem !important;
  line-height: 1.4 !important;
}
html.wall-mode .text-xl {
  font-size: 2.5rem !important;
  line-height: 1.3 !important;
}

/* 4. 取消 tracking-tight(負字距在縮放後會把字擠在一起像被切) */
html.wall-mode h1,
html.wall-mode h2 {
  letter-spacing: 0 !important;
}
```

## 四、版面規則(避免內容跨牆縫)

### 4.1 Sidebar 對齊牆縫

**Sidebar 寬度設成 1920px(剛好 1 cell)**,右邊緣自然落在第一條垂直 bezel(x=1920),內容區從 cell 2 開始,不會跨第一條牆縫。

```css
html.wall-mode aside {
  width: 1920px !important;
}
```

### 4.2 標題 / 重要文字遠離牆縫

**牆縫位置:**
- 直:x = 1920, 3840, 5760, 7680, 9600
- 橫:y = 1080, 2160

頁面 H1、KPI 數值、按鈕等**重要視覺元素應落在 cell 內部**,不要橫跨。卡片之間 grid gap 至少 64px,避免緊鄰 bezel。

```css
html.wall-mode main {
  padding: 4rem;          /* 192px,給左右各一個牆縫的呼吸區 */
}
```

### 4.3 大型 KPI 數值

獨立 cell 內顯示時,數值字應該夠大撐起該 cell 視覺重心(建議 ≥288px ≈ 6rem)。

```css
html.wall-mode .kpi-card {
  min-height: 700px;       /* 約 1.5 cell 高 */
}
html.wall-mode .kpi-card-value {
  font-size: 6rem !important;     /* 288px */
  line-height: 1 !important;
  font-weight: 700 !important;
}
```

### 4.4 表格(table)注意事項

寬表格(>5 欄)在 wall mode 容易跨 2 條以上垂直牆縫。建議:
- 列高拉大 (`line-height` 加大)
- 單列盡量在 1 cell 高內結束(避免行高 > 1080)
- 真的太寬就考慮改 **卡片網格** 取代 table,讓每筆資料在自己的 cell 裡

下表提供 wall mode(root font-size = 48px)下的具體 px 值對照:

| Tailwind class | 自訂 wall 值 | 實際 px(wall canvas) | 縮放後(1920×1080 預覽) |

| `text-lg` | 2.25rem | 108px | ≈18px |
| `text-xl` | 2.5rem | 120px | ≈20px |

## 六、Bezel Guide(開發輔助線)

開發時在 `wall-mode` 加一個高 z-index 的 overlay 顯示牆縫位置,方便看哪些元件跨了。**正式上牆前要關掉**。

```css
html.wall-mode body::before {
  content: "";
  position: absolute;
  top: 0; left: 0;
  width: 11520px;
  height: 3240px;
  pointer-events: none;
  z-index: 9999;
  background-image:
    linear-gradient(to right, transparent 1914px, rgba(255,255,255,0.85) 1914px, rgba(255,255,255,0.85) 1926px, transparent 1926px),
    linear-gradient(to right, transparent 3834px, rgba(255,255,255,0.85) 3834px, rgba(255,255,255,0.85) 3846px, transparent 3846px),
    linear-gradient(to right, transparent 5754px, rgba(255,255,255,0.85) 5754px, rgba(255,255,255,0.85) 5766px, transparent 5766px),
    linear-gradient(to right, transparent 7674px, rgba(255,255,255,0.85) 7674px, rgba(255,255,255,0.85) 7686px, transparent 7686px),
    linear-gradient(to right, transparent 9594px, rgba(255,255,255,0.85) 9594px, rgba(255,255,255,0.85) 9606px, transparent 9606px),
    linear-gradient(to bottom, transparent 1074px, rgba(255,255,255,0.85) 1074px, rgba(255,255,255,0.85) 1086px, transparent 1086px),
    linear-gradient(to bottom, transparent 2154px, rgba(255,255,255,0.85) 2154px, rgba(255,255,255,0.85) 2166px, transparent 2166px);
  background-repeat: no-repeat;
}
```

**重點:**
- 線寬 12px(縮放後 ≈ 2px,瀏覽器不會 sub-pixel 跳掉)
- 用 7 個獨立 gradient 疊加(不要用單一 gradient 多 stop,跨瀏覽器有相容問題)
- 顏色 `rgba(255,255,255,0.85)` 高對比白
- z-index 9999 在最上層,直接蓋過內容,**目的就是讓你看到「字被切」並去調整**
- 上線時用環境變數或 build flag 把這段拿掉

## 七、預覽 / 測試指南

### 在筆電上開發

1. 瀏覽器縮放維持 **100%**(不要動 Ctrl + 加減)
2. 視窗最大化或 F11 全螢幕,讓 viewport 越大越好
3. 點 Header 的 `📺 電視牆` toggle
4. 整個畫面會等比縮成 ~16.7% 顯示在筆電上(11520→1920 寬)
5. 看到 bezel 白線壓在哪些字 / 卡片上 → 那就是要修的地方

### 真牆驗證

- 主機輸出 11520×3240,瀏覽器全屏
- JS 算出 scale = 1.0,無縮放
- bezel guide 拿掉(prod build)
- 從約 3-5 公尺看,字應該可讀無壓力

## 九、已知陷阱

1. **`tracking-tight` (`letter-spacing: -0.025em`)** 在大字級縮放後字會擠成一團像被切 → wall-mode 下強制 `letter-spacing: 0`
2. **`.text-sm` 等 Tailwind class 改 font-size 時必須一起改 line-height** → 不然字溢出 line-box 跟上下行重疊
3. **`<main className="h-screen">`** 在 wall-mode 下只填 viewport 1080,而不是 body 3240 → 強制覆寫 `.h-screen { height: 100% !important; }`
4. **CSS `calc(100vw / 11520)`** 結果是 length(0.167px)不是 unitless,塞不進 `scale()` → 改用 JS 算
5. **DevTools Device 工具列上限 9999px**,沒辦法直接模擬 11520 寬;靠 JS 縮放才看得到全貌
6. **`position: fixed` 在 transform 過的祖先底下會以該祖先當 containing block** → 用 `position: absolute` 確保 bezel guide 對到 body 11520×3240 而非 viewport
