## 二、架構哲學：三層分離

本專案採用嚴格的 **關注點分離 (Separation of Concerns)** 原則，將前端程式碼組織為三個清晰的層次：

```
            User Interaction
                   │
                   ▼
        ┌──────────────────────┐
        │  Component Layer     │  純展示、Props 驅動
        │  (UI)                │  嚴禁：useEffect 抓資料、直接呼叫 API
        └──────────┬───────────┘
                   │  透過 Container 連接
                   ▼
        ┌──────────────────────┐
        │  Hook Layer          │  業務邏輯、狀態、副作用
        │  (Logic)             │  內部使用 TanStack Query / Zustand
        └──────────┬───────────┘
                   │  呼叫
                   ▼
        ┌──────────────────────┐
        │  Service Layer       │  純資料存取（fetch/axios）
        │  (API)               │  嚴禁：React Hooks、UI 邏輯
        └──────────┬───────────┘
                   │
                   ▼
            Backend / Mock Data
```

### 2.1 Component Layer（UI 展示層）

**職責**：
- 純展示，Props 驅動
- 渲染 JSX，處理使用者互動事件（呼叫傳入的 callback）

**嚴禁**：
- ❌ `useEffect` 用於抓資料
- ❌ 直接呼叫 Service
- ❌ 複雜業務邏輯

**範例**：
```tsx
// components/Dut/DutList.tsx
type Props = {
  duts: Dut[];
  isLoading: boolean;
  onView: (dut: Dut) => void;
  onDelete: (id: string) => void;
};

export function DutList({ duts, isLoading, onView, onDelete }: Props) {
  if (isLoading) return <Skeleton className="h-64" />;
  if (duts.length === 0) return <EmptyState message="尚未新增 DUT" />;
  return (
    <Table>
      {duts.map((dut) => (
        <TableRow key={dut.id}>
          <TableCell>{dut.name}</TableCell>
          <TableCell><StatusBadge status={dut.status} /></TableCell>
          <TableCell>
            <Button onClick={() => onView(dut)}>詳情</Button>
          </TableCell>
        </TableRow>
      ))}
    </Table>
  );
}
```

### 2.2 Hook Layer（業務邏輯層）

**職責**：
- 管理狀態、副作用、業務邏輯
- 呼叫 Service
- 內部可使用 TanStack Query（不違反分層，Query 只管 server state cache）
- 只回傳 View 需要的資料與函數

**範例**：
```tsx
// hooks/Dut/useDutList.ts
export function useDutList(filters: DutFilters) {
  const query = useQuery({
    queryKey: ['dut', 'list', filters],
    queryFn: () => dutService.list(filters),
    staleTime: 30_000,
  });
  return {
    duts: query.data?.items ?? [],
    total: query.data?.total ?? 0,
    isLoading: query.isLoading,
    error: query.error,
    refresh: query.refetch,
  };
}
```

### 2.3 Service Layer（資料存取層）

**職責**：
- 處理所有 HTTP 請求
- 定義 API 端點
- 回傳 Promise 與型別化資料

**嚴禁**：
- ❌ React Hooks
- ❌ UI 邏輯、狀態管理

**範例**：
```tsx
// services/Dut/dutService.ts
export const dutService = {
  async list(filters: DutFilters): Promise<Paginated<Dut>> {
    const { data } = await apiClient.get('/dut', { params: filters });
    return data;
  },
  async get(id: string): Promise<Dut> {
    const { data } = await apiClient.get(`/dut/${id}`);
    return data;
  },
  async create(input: DutInput): Promise<Dut> {
    const { data } = await apiClient.post('/dut', input);
    return data;
  },
  // ...
};
```

### 2.4 Container 模式

每個業務模組有一個 `XxxContainer.tsx`，負責：
- 組合多個 hooks
- 管理本地 UI 狀態（如 Dialog 開關）
- 把 props 傳給純 UI 元件

```tsx
// components/Dut/DutManagementContainer.tsx
export function DutManagementContainer({ dutType }: { dutType: DutType }) {
  const { duts, isLoading, refresh } = useDutList({ type: dutType });
  const { create, isCreating } = useDutForm();
  const { testInterface, isTesting } = useDutInterfaceTest();
  const [selected, setSelected] = useState<Dut | null>(null);
  const [formOpen, setFormOpen] = useState(false);

  return (
    <>
      <PageHeader title={`連接介面驗證 - ${dutType}`}>
        <Button onClick={() => setFormOpen(true)}>新增設備</Button>
      </PageHeader>
      <DutList
        duts={duts}
        isLoading={isLoading}
        onView={setSelected}
      />
      <DutFormDialog
        open={formOpen}
        onClose={() => setFormOpen(false)}
        onSubmit={async (input) => {
          await create(input);
          await refresh();
          setFormOpen(false);
        }}
        isSubmitting={isCreating}
      />
      <DutDetailDialog
        dut={selected}
        onClose={() => setSelected(null)}
        onTestInterface={testInterface}
        isTesting={isTesting}
      />
    </>
  );
}
```

### 目錄結構規範

| 規範 | 說明 |
|------|------|
| **功能導向分組** | `services/Dut/`, `hooks/Dut/`, `components/Dut/` —— 同一個功能的三層集中在各自目錄 |
| **shadcn/ui 獨立** | `components/ui/` 不與業務混合，可在任何地方重用 |
| **共用元件** | 跨模組使用的放 `components/common/` |
| **Mock 雙軌** | 每個 Service 配一個 `mockXxxService.ts` |
| **統一匯出** | `services/index.ts` 中根據 env 決定匯出 Mock 或正式版 |

### Group 用途

- `(auth)`：登入相關，無 Sidebar
- `(dashboard)`：登入後的所有頁，共用 AppShell layout

### 5.1 Sidebar 三層階層（與 Figma 對齊）

Sidebar 分三層：**頂層群組 → 中層容器（DUT/apps）→ 葉節點類型**。每層都可透過 `uiStore.toggleNavItem(id)` 獨立展開／收合；葉節點才帶 `href`，中層與群組僅作為 toggle。UI 文案以 Figma 為準（例：「資料品質驗證」而非「資料基礎驗證」）。

```
[leaf]    📊 總覽                          → /overview
[group]   🔌 連接介面驗證 (interface-validation)
  [mid]   └─ DUT (interface-dut)
  [leaf]     ├─ SMO                         → /interface-validation/smo
  [leaf]     ├─ RIC                         → /interface-validation/ric
  [leaf]     ├─ xApp                        → /interface-validation/xapp
  [leaf]     └─ rApp                        → /interface-validation/rapp
[group]   📁 資料品質驗證 (data-validation)
  [mid]   └─ DUT (data-dut)
  [leaf]     ├─ SMO                         → /data-validation/smo
  [leaf]     └─ RIC                         → /data-validation/ric
[group]   🧠 智慧程度驗證 (intelligence-validation)
  [mid]   └─ DUT (intelligence-apps)
  [leaf]     ├─ xApp                        → /intelligence-validation/xapp
  [leaf]     └─ rApp                        → /intelligence-validation/rapp
[leaf]    🎯 端對端測試情境                → /test-scenarios
[group]   🌍 場域管理 (site-management)
  [leaf]   ├─ 國內場域                      → /site-management/domestic
  [leaf]   └─ 國外場域                      → /site-management/international
```

> **命名與路由分離原則**：路由 path 仍使用英文 kebab-case `data-validation`（對齊後端 strategy key `data_validation`），不隨 UI 文案改動。改文案只動 Sidebar `label`、`PageHeader` 標題、i18n key。

**DutList 欄位定義**（9 欄,對齊 Figma):

| 欄位 | 內容 | 備註 |
|---|---|---|
| 設備資訊 | `name`(粗體) + `id`(灰色小字) | 兩行式 |
| 類型 | `type` outline Badge | SMO/RIC/xApp/rApp |
| Endpoint | `endpoint` `font-mono text-xs` | |
| 部署場域 | `site_name` + `site_environment` Badge | 由 Serializer 從 FK 帶出,不是 DUT 自己的欄位 |
| 狀態 | `<StatusBadge status={dut.status}/>` | 單一 status(online/offline/error) |
| 資料格式 | `data_format` 純文字 | 若空則顯示 `-` |
| 回應時間 | `response_time_ms` + 顏色 | <200 綠 / <1000 橘 / ≥1000 紅 / null 顯示 `-` |
| 最後檢查 | `formatDate(last_check)` | |
| 操作 | icon-only Button:🔄 刷新 / 👁 檢視 / ✏ 編輯 | 刷新=重用 `POST /api/dut/{id}/test-interface/`,只轉動該列的 icon |

**DutFormDialog 欄位**:

## 八、WebSocket 整合

WebSocket 訂閱放在 Hook 層，對 Component 暴露為一般的 state：

```typescript
// hooks/ValidationRun/useRunStream.ts
export function useRunStream(runId: string | null) {
  const [state, setState] = useState<ValidationRun | null>(null);
  const [logs, setLogs] = useState<LogEntry[]>([]);
  const [metrics, setMetrics] = useState<Record<string, number>>({});

  useEffect(() => {
    if (!runId) return;
    const ws = new WebSocket(`${process.env.NEXT_PUBLIC_WS_BASE}/runs/${runId}`);
    ws.onmessage = (ev) => {
      const msg: WsMessage = JSON.parse(ev.data);
      switch (msg.type) {
        case 'progress': setState(msg.payload); break;
        case 'log': setLogs((prev) => [...prev, msg.payload]); break;
        case 'metric': setMetrics((prev) => ({ ...prev, [msg.payload.key]: msg.payload.value })); break;
        case 'completed': setState((prev) => prev ? { ...prev, ...msg.payload } : null); break;
      }
    };
    return () => ws.close();
  }, [runId]);

  return { state, logs, metrics };
}
```

## 十三、開發規範總結

### ✅ 必須遵守

- Service 使用 async/await 回傳 Promise
- Hook 使用 TanStack Query 包 server state
- Component 只接收 Props，不處理業務邏輯
- 每個業務模組有自己的 Container
- 所有 API 呼叫都有 try-catch 與錯誤 toast
- 使用 TypeScript strict 模式

### ❌ 嚴禁

- Component 直接 import Service
- Service 使用 React Hooks
- 跨層混合職責
- 忽略 API 錯誤
- 使用 `any`（確有需要用 `unknown`）
